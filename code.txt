---
I chose **Typer** over `argparse` because it provides a cleaner and more maintainable way to build command-line applications. It uses Python type hints, automatically generates help messages, and requires less boilerplate code. This keeps the CLI implementation small and easy to extend while remaining appropriate for the scope of this project.
---
The assignment explicitly excludes using a database, making JSON a suitable persistence format.
Benefits:
* Simple to implement.
* Human-readable and easy to inspect.
* Requires no additional dependencies or setup.
* Sufficient for a small number of alarms.
Trade-offs:
* Entire file must be read and written for updates.
* Not suitable for concurrent access or large datasets.
These limitations are acceptable given the project's intended scale.
---
The scheduler checks for due alarms once every second instead of using an event-driven or asynchronous scheduler.
Benefits:
* Straightforward implementation.
* Easy to understand and debug.
* Portable across operating systems.
* No external scheduling libraries required.
Trade-offs:
* Small amount of unnecessary CPU usage.
* Alarm precision is limited to the polling interval.
For a lightweight CLI alarm application, this trade-off is acceptable.
---
The `Alarm` model is implemented as a Python `dataclass` because it represents structured data with minimal business logic.
Benefits:
* Less boilerplate than a traditional class.
* Automatically generates constructors and string representations.
* Improves readability.
* Easy to serialize to and from JSON.
Using a dataclass keeps the data model concise and maintainable.
---
Business logic is separated from the CLI through a service layer (`commands.py`).
Responsibilities are divided as follows:
* **CLI**: Parses user input.
* **Service layer**: Validates requests and applies business rules.
* **Storage**: Handles persistence.
This separation improves maintainability, makes the code easier to test, and allows future interfaces (such as a GUI or REST API) to reuse the same business logic.
---
A one-second polling interval provides a good balance between responsiveness and simplicity.
Benefits:
* Alarms trigger within approximately one second of their scheduled time.
* Minimal CPU usage.
* Easy to reason about and test.
Since alarm clocks do not require millisecond precision, increasing implementation complexity for higher timing accuracy was not justified.
---
AI tools (ChatGPT) were used during development to assist with:
* Refining ambiguous requirements.
* Discussing project architecture.
* Brainstorming implementation approaches.
* Generating initial boilerplate code.
* Reviewing design decisions.
* Improving project documentation.
All AI-generated suggestions were manually reviewed, adapted where necessary, and validated through implementation and testing. Final design decisions and code organization were based on engineering judgment rather than accepting generated output without review.
---
Because the assignment intentionally leaves several requirements unspecified, the following assumptions were made:
* The application is intended for a single local user.
* Alarms are stored locally in a JSON file.
* The scheduler must be running (`alarm run`) for alarms to trigger.
* Alarm times use the system's local time.
* Only minute-level scheduling is required.
* Alarm notifications are delivered through the terminal using a terminal bell and a printed message.
* Multiple alarms may share the same trigger time.
* A snoozed alarm is postponed by five minutes.
---
If this project were expanded beyond the assignment, potential improvements include:
* Desktop notifications.
* Custom alarm sounds.
* Time zone support.
* Weekly or custom recurring schedules.
* Editing existing alarms.
* Better error handling and input validation.
* SQLite-based persistence for larger datasets.
* Background service/daemon mode.
* Import/export functionality.
* Additional automated tests, including integration and end-to-end testing.
